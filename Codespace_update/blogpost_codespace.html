DIGGS Codelist Reference Update: A Unified Approach for All Users

Dear DIGGS Community,

We're excited to share an important update to how DIGGS handles references to codeTypes. This change will make our data exchange standard more robust and consistent across all applications.

What's Changing?

Going forward, when referencing a codeType in DIGGS, we will require a specific format:

  • A reference must include a hashtag (#) followed by the DIGGS name
  • Any additional text after this reference is optional and considered uncontrolled metadata
  • This optional text can be used for things like language translations or application-specific mappings
  • Importantly, this optional text is no longer part of the validation process

Before and After Example:

Previous approach (no longer valid):

<rolePerformed codeSpace="https://diggsml.org/def/codes/DIGGS/0.1/roles.xml">helper</rolePerformed>

New unified approach:

<rolePerformed codeSpace="https://diggsml.org/def/codes/DIGGS/0.1/roles.xml#helper">DrillingAssistant</rolePerformed>

In this example, "helper" is the standardized DIGGS code, while "DrillingAssistant" is optional text that can provide additional context or translation.

Why This Matters: Real-World Examples

Example 1: International Project Collaboration

Imagine you're working on a dam monitoring project with international partners. Your team in the US uses the term "Driller" while your German colleagues use "Bohrarbeiter" for the same role. With the new approach:

<rolePerformed codeSpace="https://diggsml.org/def/codes/DIGGS/0.1/roles.xml#driller">Bohrarbeiter</rolePerformed>

The validation focuses on "#driller" (the standard code), while "Bohrarbeiter" provides human-readable context. This ensures both technical accuracy and clear communication.

Example 2: Piezometer Data Integration

When integrating piezometer data from multiple vendors using different terminology:

<measurand codeSpace="https://diggsml.org/def/codes/DIGGS/0.1/properties.xml#water_depth">PoreWaterPressure</measurand>

The standard "#water_depth" ensures correct data integration, while "PoreWaterPressure" preserves the vendor's terminology.

Benefits of This Change

  1. Consistency: One unified approach for all reference scenarios
  2. Flexibility: Supports translations, mapping to proprietary terms, and additional context
  3. Reliable Validation: Clear separation between the required standard code and optional metadata
  4. Future-Proofing: Better supports evolution of the standard while maintaining backward compatibility

What This Means for You

  • Existing Files: Older DIGGS files will need to be updated to this new format
  • Tools & Validators: Updates are in progress for validators and codelist reference generation tools
  • Documentation: We are updating our documentation to reflect this change

While this change will impact backward compatibility, it's a necessary step toward a more robust and flexible standard that will serve the geotechnical community better in the long run.
