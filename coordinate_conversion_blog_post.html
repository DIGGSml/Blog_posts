The Root Cause: Inconsistent Coordinate Reference Systems

Our analysis reveals that the georeferencing crisis stems from three primary issues:

  1. Project-Specific Stationing Systems: The most common issue involves boring logs referenced only by project stationing (e.g., "STA 125+50, 85' LT") where the stationing baseline is completely arbitrary to that specific project and the survey control data needed for conversion is not stored with the boring log data.
  2. Missing EPSG References: Legacy boring logs contain coordinates without specifying their coordinate reference system. A coordinate pair like "567,890.5, 123,456.7" could represent State Plane, UTM, or any number of local coordinate systems.
  3. Inconsistent Coordinate System Usage: Within a single project, different consultants may use different coordinate systems without proper documentation.
  4. Format Variations: Coordinates appear in decimal degrees, degrees-minutes-seconds, feet, meters, and various state plane zones without consistent identification.

Consider this real example from our Alabama DOT dataset processing:

<!-- Legacy DIGGS boring log with station/offset reference -->
<samplingFeature>

        <referencePoint>

This real ALDOT boring log shows the exact problem: coordinates stored as station/offset values (55.0, -85.0) relative to an undefined survey centerline. Without an EPSG code reference and the corresponding survey control, this boring log cannot be reliably positioned on any map or integrated with other datasets. This represents one of the 34,322 ALDOT boring logs that couldn't be georeferenced‚Äî$102.9 million in lost taxpayer-funded investigations.

The DIGGS 2.7 Solution: Mandatory EPSG Validation

I'm excited to announce that with the upcoming release of DIGGS version 2.7, this georeferencing crisis will finally be addressed through schema-level enforcement. DIGGS 2.7 introduces mandatory EPSG code validation for all location elements, ensuring that every boring log includes unambiguous spatial reference information.

What Changes in DIGGS 2.7

The new DIGGS 2.7 schema requires that all coordinate references include a mandatory srsName attribute with a valid EPSG code:

<!-- DIGGS 2.7 compliant coordinate reference -->
<samplingFeature>
    <Borehole gml:id="Borehole_IM_I065_390_BH1">
        <gml:name>BH-1</gml:name>
        <referencePoint>
            <PointLocation gml:id="Point_Location_STA_55_00" srsDimension="3" 
                          srsName="http://www.opengis.net/def/crs/EPSG/0/26929">
                <gml:pos>654321.45 1234567.89 287.5</gml:pos>
            </PointLocation>
        </referencePoint>
        <centerLine>
            <LinearExtent gml:id="Linear_Extent_0" srsDimension="3" 
                         srsName="http://www.opengis.net/def/crs/EPSG/0/26929">
                <gml:posList>654321.45 1234567.89 287.5 654321.45 1234567.89 265.0</gml:posList>
            </LinearExtent>
        </centerLine>
        <gml:description>IM-I 065 (390) SOIL SURVEY, STA. 55+00, 85' LT OF SURVEY C/L</gml:description>
    </Borehole>
</samplingFeature>

With EPSG:26929 (Alabama State Plane East, NAD83) explicitly specified in the srsName attribute, the same ALDOT boring log now contains actual geographic coordinates instead of station/offset values. Processing systems can reliably convert these coordinates to any other coordinate system, integrate them with regional datasets, and display them on web maps. The station/offset description is preserved while the coordinates provide unambiguous spatial reference.

This simple change will revolutionize geotechnical data interoperability. Going forward, our profession can reliably georeference any DIGGS file in any coordinate system, enabling:

  • Seamless data sharing between jurisdictions using different coordinate systems
  • Automatic coordinate conversion for web mapping and GIS applications
  • Reliable integration of multi-state project datasets
  • Machine learning training on properly georeferenced datasets

Legacy Data Recovery Process

For datasets like ALDOT's station/offset boring logs, the recovery process involves:

  1. Survey Control Recovery: Locate original highway alignment and survey control data
  2. Manual Coordinate Calculation: Convert station/offset to state plane coordinates using survey data
  3. API Validation & Conversion: Use our API to verify coordinates and convert between systems if needed
  4. DIGGS 2.7 File Creation: Generate compliant files with proper srsName attributes

Cross-Jurisdictional Project Integration

The API's primary strength is enabling seamless data sharing between jurisdictions using different coordinate systems:

  • Virginia DOT provides data in State Plane North (EPSG:2283)
  • Maryland DOT uses State Plane coordinates (EPSG:2248)
  • API converts all data to WGS84 (EPSG:4326) for web visualization
  • Result: Unified I-95 corridor dataset for regional analysis

Multi-State Project Integration

Consider a typical interstate highway improvement project crossing Virginia, Maryland, and Pennsylvania. Previously, each state's boring logs would use different coordinate systems:

  • Virginia: State Plane North (EPSG:2283) in US Survey Feet
  • Maryland: State Plane (EPSG:2248) in Meters
  • Pennsylvania: State Plane South (EPSG:2272) in US Survey Feet

With DIGGS 2.7 compliance and our coordinate conversion API, the project workflow becomes:

  1. Data Collection: Each state provides DIGGS 2.7 files with mandatory EPSG references
  2. Coordinate Standardization: All coordinates are automatically converted to WGS84 for web mapping
  3. Unified Analysis: Regional subsurface modeling uses consistently referenced spatial data
  4. Visualization: Interactive maps display boring logs from all three states in a unified coordinate system
